# Roadmap — from grounded orchestrator to a brain-shaped knowledge system

North star, in one paragraph: orc maps a codebase into a living knowledge graph, lets
agents persist and retrieve memory efficiently instead of re-reading the world, keeps
plans as first-class editable graphs a human can annotate and approve, updates the
visual graph live while agents run, and keeps a replayable audit feed of every
individual step and conversation. Token safety comes from the shape itself: the complex
web a project normally scatters across chats and heads is visualized, event-sourced, and
retrievable across sessions — decisions and research included. The horizon is a neuronal
lifecycle: access patterns (hot paths) drive how knowledge is ranked, decayed, archived,
and restored — housekeeping the way a brain does it, tuned from measured traffic, never
from round numbers.

**Two doors, one substrate.** Interactive: a human works the webchat (later: TUI) — the
guided flow with analyze gates, plan review, annotation, approval. Headless: an external
agent (Claude Code, any MCP client) goes full-auto and uses the system BEHIND the flow —
durable persistence, replayable logs, plan graphs, knowledge memory, verified artifacts.
Same kernel, same events, same guarantees; the door only changes who is driving and where
the autonomy dial sits. orc is the agent's external brain and execution ledger — whether
the agent is guided by a human or is one.

## Next — concrete, not yet scheduled

**Scheduling (pillar 9).** Time-based runs: "analyze nightly", "regenerate docs weekly",
"retry blocked tasks at 9am". The durable-execution layer (DBOS Transact) ships scheduled
workflows — verify the API surface of our pinned version when this starts; the seam is the
DBOS port, nothing above it changes. A scheduled run is an ordinary `run_started` with a
`source: schedule` attribution — the audit story stays whole. Gate: recurring real usage,
i.e. the first time a human runs the same request three times by hand.

**Collaboration prerequisites (pillar 10).** Honest sequence, no big-bang multi-user:
1. **Attribution first** (starts in P7): every mutation carries who drove — cli | web |
   mcp | schedule. Events already carry author patterns in memory writes; extend to the
   action surface.
2. **Identity**: named human actors in events (`approvedBy` today is a constant string).
3. **Transport**: today's server is 127.0.0.1 + CSRF token, single project fence — correct
   for one user. Multi-user needs authn and per-user approval records. Do NOT relax
   localhost before identity exists.
Gate: a second human actually needs to approve/annotate on the same project.

## Horizon — the neuronal system (pillar 11)

The full design exists and was deliberately deferred with reasoning intact:
`docs/IDEAS.md` entry 1 + `docs/superpowers/specs/2026-07-19-neuron-memory-web-research-design.md`.
Do not rebuild it early; the discipline that deferred it is the same discipline that makes
it good when it fires.

What the vision adds ("track hot paths like a brain stores and retrieves"):

- **Measurement layer — DONE as of 2026-07-22.** `memory_accessed` events are canonical
  (survive read-model rebuilds), now task/step-bound in the envelope, surfaced as
  `hits`/`lastAccessedAt` in `orc memory ls` and per-task pulls in the UI. Access is
  recorded only on delivery (a hit), never on a miss or a summary listing — the signal is
  honest traffic, exactly what a decay policy must be tuned against.
- **Trigger (unchanged, now observable):** `memory_search` regularly surfaces stale notes
  ahead of current ones AND the hits distribution shows a clear hot/cold split. Watch it
  with `orc memory ls` as graphs grow past a few hundred notes.
- **When it fires:** activation strength from observed access (not authored importance),
  half-life fitted to the measured distribution (not 30 because 30 sounds right),
  dormant notes leave default search/traversal, an explicit depth-controlled sweep
  retires them to a restorable Markdown archive. Ranker graded against independent
  ground truth (what agents actually read), never against the graph's own `supersedes`
  edges — that grading is circular and IDEAS entry 1 documents why.
- **This is the housekeeping story:** hot paths stay cheap to pull (token safety
  compounds), cold knowledge leaves the working set without being destroyed, and the
  archive is replayable like everything else.

## Explicitly not on the roadmap

- **Generic framework packaging** — revisit only after P7 proves the core seams with a
  second consumer, and only if an external project actually wants the packages.
- **Lifecycle policy before its trigger** — see Horizon; constants come from measurement.
- **Multi-user transport before identity** — see Next.

## Rules

Same as the flawless plan: every phase evidence-backed, one phase per PR, acceptance
checks land as tests. Horizon items are gated by TRIGGERS, not dates — the roadmap names
the trigger and the watch procedure; the trigger firing is what schedules the work.
