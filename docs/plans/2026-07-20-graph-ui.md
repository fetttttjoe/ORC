# Plan: live visual map of the orc graph — ports & adapters, built to scale to big graphs and a future TUI

A local web page (`orc graph` → http://127.0.0.1:7749) rendering the full execution + knowledge
graph (tasks, steps, artifacts, memory notes) as an interactive WebGL force graph that grows
live, with a per-node detail panel and a project switcher ("sessions" = projects). All logic
lives in a transport-free core package; the web server, the browser renderer, and the future
TUI are thin adapters over it.

Everything is a projection of the Postgres event log — no SurrealDB, no new database, no
separate build tooling (Bun bundles the page).

## Rendering library — research + verdict

Checked against npm 2026-07 (all MIT):

| Library | Latest | Model | Realistic ceiling | Notes |
|---|---|---|---|---|
| **sigma** | 3.0.3 (active) | WebGL renderer over a **graphology** graph | ~50–100k nodes | Layout is separate (ForceAtlas2 in a **web worker** — main thread stays free). Incremental by design: mutate the graphology graph, sigma re-renders. Mature picking/hover/camera. |
| graphology | 0.26.0 | graph data model + algorithms | n/a | Doubles as the client-side graph store — our patches apply as graph mutations. |
| @cosmograph/cosmos | 2.0.0-**beta**.20 | GPU (regl) layout + render | 100k–1M | Raw performance champion, but beta and coarser interactions. The upgrade target, not the start. |
| force-graph / 3d-force-graph | 1.51 / 1.80 | canvas / three.js, d3-force on main thread | ~5–20k | Easiest API; layout chokes first. Fine today, wrong for "a lot of nodes". |
| @antv/g6 | 5.1.1 | full graph framework | ~10k+ | Far more framework than we need. |

**Verdict: sigma + graphology + graphology-layout-forceatlas2 (worker).** WebGL rendering,
worker layout, incremental mutations that map 1:1 onto our patch protocol, mature events.
The browser side hides it behind a `GraphRenderer` adapter (below), so when a graph outgrows
ForceAtlas2 (~100k nodes), a `CosmosRenderer` slots in without touching anything else.

## Architecture — ports & adapters

```
                 ┌────────────────────────────────────────────┐
                 │ packages/ui-core   (NO http, NO dom, NO tui)│
                 │  • buildGraph / diffGraphs / GraphPatch     │
                 │  • nodeDetail(session, nodeId)              │
                 │  • ProjectSessions: event log → live        │
                 │    {state, graph, seq} + patch subscribers  │
                 │  • (later) OrcActions command port          │
                 └────────┬──────────────────┬────────────────┘
     imports directly     │                  │ imports directly
                ┌─────────▼────────┐  ┌──────▼─────────┐   ┌──────────────┐
                │ packages/graph-ui│  │ future: TUI     │   │ CLI (today)  │
                │ Bun.serve adapter│  │ adapter (same   │   │ stays as-is; │
                │ JSON + SSE       │  │ subscriptions,  │   │ migrates to  │
                └─────────┬────────┘  │ no HTTP needed) │   │ OrcActions   │
                          │           └────────────────┘   │ incrementally│
                ┌─────────▼─────────────────┐              └──────────────┘
                │ browser: GraphRenderer     │
                │ adapter → SigmaRenderer    │
                │ (later CosmosRenderer)     │
                └───────────────────────────┘
```

Design rules locked up front:
- **ui-core imports only `@orc/contracts` + `@orc/kernel`.** No `Bun.serve`, no DOM types, no
  renderer. Enforced by tsc project references — a TUI gets everything by importing ui-core,
  never by scraping the web server.
- Live updates are **incremental from day one**: subscribers receive per-event `GraphPatch`es
  (`addNodes/updateNodes/removeNodeIds/addLinks/removeLinks`); a single new note is a single
  small message; client work is O(patch), never O(graph).
- One projection, no drift: `buildGraph` is the ONLY events→graph function; patches are always
  `diffGraphs(prev, next)` — never a second hand-written per-event transition. Full load and
  patch stream cannot disagree.
- Lossless resume is platform-native: SSE messages carry `id: <seq>`; browser reconnects send
  `Last-Event-ID`; first attach passes `?fromSeq=` from the snapshot's `seq`; the server
  resumes via the log's own `subscribe({ fromSeq })` catch-up.
- Read-only server, bound to 127.0.0.1. All DOM via `textContent`/`createElement` — never
  `innerHTML` (note bodies are agent-written).
- **Command port (`OrcActions`) is designed now, implemented when the first mutating UI
  element lands** (approve/reply/cancel buttons). Shape: the exact operations `buildProgram`
  wires today (`approve(taskId)`, `cancel(taskId)`, `reply(taskId, text)`,
  `propose(taskId, opts)`, `run(taskId, cwd)`, memory CRUD), defined in ui-core, implemented
  by wrapping the same kernel/ExecutionPort calls `packages/cli/src/main.ts` uses. Web adds
  POST routes over it; TUI calls it directly; CLI migrates command-by-command. Not stubbed in
  v1 — no dead interfaces.
- `ponytail:` patch derivation refolds per event — O(events) per event, fine at current scale;
  the upgrade (incremental `applyEvent` exported from the kernel fold) changes only
  `ProjectSessions` internals — protocol and clients are already incremental.

Deferred (explicitly out of v1): `CosmosRenderer` (>100k nodes), the TUI adapter itself,
`OrcActions` implementation + POST routes, 3D mode, time scrubbing by seq, Surreal neighbor
ranking in the panel, unifying kernel `foldPlanNotes` onto the shared fold, a project-name
registry event (non-cwd projects show short ids), incremental kernel fold, viewport culling.

### Phase 1 — shared seams (small touches to existing packages, all test-covered)

1. **Step 1.1 — extract the live-notes fold into contracts**
   - **Why:** `sweep.ts` already folds `memory_written`/`memory_deleted` into a last-write-wins
     live map; ui-core needs the identical fold (plus note bodies). One shared fold instead of
     a third copy (kernel's `foldPlanNotes` is a second; unifying it is deferred).
   - **Files:** `packages/contracts/src/memory.ts` (add; `EventRecord`/`EVENT_KIND` come from
     `./events` — extend the imports at the top as needed).
     ```ts
     export const noteKey = (scope: string, id: string): string => `${scope}\u0000${id}`

     // Last-write-wins fold of the memory event stream into the live note set. Lenient like
     // fold(): an unparseable payload is skipped, never applied. Input needs only kind+payload
     // so both EventRecord[] and trimmed test fixtures work.
     export function foldLiveNotes(
       events: Array<Pick<EventRecord, 'kind' | 'payload'>>,
     ): Map<string, MemoryWrittenPayload> {
       const live = new Map<string, MemoryWrittenPayload>()
       for (const e of events) {
         if (e.kind === EVENT_KIND.memory_written) {
           const p = MemoryWrittenPayload.safeParse(e.payload)
           if (p.success) live.set(noteKey(p.data.note.scope, p.data.note.id), p.data)
         } else if (e.kind === EVENT_KIND.memory_deleted) {
           const p = MemoryDeletedPayload.safeParse(e.payload)
           if (p.success) live.delete(noteKey(p.data.scope, p.data.id))
         }
       }
       return live
     }
     ```
   - **Verify:** `bunx tsc --noEmit -p packages/contracts` clean.

2. **Step 1.2 — rewrite `orphanedNotes` on top of the shared fold**
   - **Why:** deletes the duplicate fold; sweep semantics (adoption, survivor references,
     cli-author immunity, lenient parse) stay identical and stay covered by `sweep.test.ts`.
   - **Files:** `plugins/memory/src/sweep.ts` (full body becomes):
     ```ts
     import { foldLiveNotes, noteKey, type EventRecord } from '@orc/contracts'

     export interface SweptNote { id: string; scope: string; title: string; summary: string }

     // Deterministic cancel-time GC — the orchestrator decides what stays vs goes, not an agent.
     // A note GOES iff its LAST writer's task is in `cancelled` AND no surviving note links to
     // it. Adopted ids (re-written by a live writer) stay; cli/null-authored notes are never
     // swept. Deletion is a memory_deleted event, so content stays in the log for audit/replay.
     export function orphanedNotes(events: EventRecord[], cancelled: Set<string>): SweptNote[] {
       const notes = [...foldLiveNotes(events).values()].map(({ note, author }) => ({
         scope: note.scope, id: note.id, title: note.title, summary: note.summary,
         links: note.links.map(l => l.id), taskId: author.taskId ?? '',
       }))
       const referenced = new Set(
         notes.filter(n => !cancelled.has(n.taskId)).flatMap(n => n.links.map(id => noteKey(n.scope, id))),
       )
       return notes
         .filter(n => cancelled.has(n.taskId) && !referenced.has(noteKey(n.scope, n.id)))
         .map(({ id, scope, title, summary }) => ({ id, scope, title, summary }))
     }
     ```
     `MemoryWrittenPayload`'s note schema defaults `scope` to `'project'` and `links` to `[]`,
     so the trimmed fixtures in `sweep.test.ts` keep parsing.
   - **Verify:** `bun test plugins/memory/src/sweep.test.ts packages/cli/src/main.test.ts` —
     all pass unchanged (3 sweep tests + cancel-sweep wiring test).

3. **Step 1.3 — `listProjectIds` in the kernel storage layer**
   - **Why:** projects exist only as distinct `project_id` values in `events`; `PostgresStore`
     is deliberately project-bound, so discovery is a standalone helper next to it (kernel owns
     Postgres; viewers must not open their own driver).
   - **Files:** `packages/kernel/src/storage/postgres.ts` (append), `packages/kernel/src/storage/index.ts`
     (re-export: `export { PostgresStore, listProjectIds, type Tx } from './postgres'`).
     ```ts
     // Cross-project discovery for read-only viewers. Deliberately outside PostgresStore,
     // which is bound to ONE project by design.
     export async function listProjectIds(url: string): Promise<string[]> {
       const pool = new pg.Pool({ connectionString: url, max: 1 })
       try {
         const r = await pool.query<{ project_id: string }>(
           'SELECT project_id, MAX(seq) AS last FROM events GROUP BY project_id ORDER BY last DESC',
         )
         return r.rows.map(row => row.project_id)
       } finally {
         await pool.end()
       }
     }
     ```

5. **Step 2.2 — projection: `buildGraph`, `diffGraphs`, `GraphPatch`**
   - **Why:** the one events→graph function and the one patch derivation; everything else in
     every adapter is plumbing around these.
   - **Files:** `packages/ui-core/src/graph.ts`
     ```ts
     import { foldLiveNotes, noteKey, type EventRecord } from '@orc/contracts'
     import type { State } from '@orc/kernel'

     export interface GraphNode {
       id: string
       type: 'task' | 'step' | 'artifact' | 'note'
       label: string
       detail: string // task: status; step: run status or 'pending'; artifact: size; note: kind
     }
     export interface GraphLink { source: string; target: string; type: string }
     export interface Graph { nodes: GraphNode[]; links: GraphLink[] }
     export interface GraphPatch {
       addNodes: GraphNode[]
       updateNodes: GraphNode[]     // same id, changed label/detail (e.g. step status flip)
       removeNodeIds: string[]
       addLinks: GraphLink[]
       removeLinks: GraphLink[]
     }
     export const emptyPatch = (p: GraphPatch): boolean =>
       !p.addNodes.length && !p.updateNodes.length && !p.removeNodeIds.length && !p.addLinks.length && !p.removeLinks.length

     export const stepNodeId = (taskId: string, id: string) => `step:${taskId}:${id}`
     export const noteNodeId = (scope: string, id: string) => `note:${noteKey(scope, id)}`

     export function buildGraph(state: State, events: EventRecord[]): Graph {
       const nodes: GraphNode[] = []
       const links: GraphLink[] = []
       for (const t of state.tasks.values()) {
         nodes.push({ id: t.id, type: 'task', label: t.title, detail: t.status })
         if (t.parentId && state.tasks.has(t.parentId)) links.push({ source: t.parentId, target: t.id, type: 'child' })
         const plans = state.plans.get(t.id)
         const plan = plans?.approvedVersion != null ? plans.versions[plans.approvedVersion - 1] : undefined
         for (const s of plan?.steps ?? []) {
           const run = state.steps.get(t.id)?.get(s.id)
           nodes.push({ id: stepNodeId(t.id, s.id), type: 'step', label: s.title, detail: run?.status ?? 'pending' })
           links.push({ source: t.id, target: stepNodeId(t.id, s.id), type: 'plan' })
           for (const dep of s.dependsOn) links.push({ source: stepNodeId(t.id, dep), target: stepNodeId(t.id, s.id), type: 'depends' })
         }
         for (const a of state.artifacts.get(t.id) ?? []) {
           const id = `artifact:${t.id}:${a.path}`
           nodes.push({ id, type: 'artifact', label: a.path, detail: `${a.size}B` })
           links.push({ source: a.stepId ? stepNodeId(t.id, a.stepId) : t.id, target: id, type: 'out' })
         }
       }
       const live = foldLiveNotes(events)
       for (const { note, author } of live.values()) {
         nodes.push({ id: noteNodeId(note.scope, note.id), type: 'note', label: note.title, detail: note.kind })
         if (author.taskId && state.tasks.has(author.taskId))
           links.push({ source: author.taskId, target: noteNodeId(note.scope, note.id), type: 'wrote' })
         for (const l of note.links)
           if (live.has(noteKey(note.scope, l.id))) // links resolve same-scope, system-wide semantics
             links.push({ source: noteNodeId(note.scope, note.id), target: noteNodeId(note.scope, l.id), type: l.kind })
       }
       return { nodes, links }
     }

     const linkKey = (l: GraphLink) => `${l.source}\u0000${l.target}\u0000${l.type}`

     // Pure id-keyed diff between two projections — the ONE way patches are derived, so the
     // patch stream can never drift from what a fresh full load would return.
     export function diffGraphs(prev: Graph, next: Graph): GraphPatch {
       const prevN = new Map(prev.nodes.map(n => [n.id, n]))
       const nextN = new Map(next.nodes.map(n => [n.id, n]))
       const prevL = new Map(prev.links.map(l => [linkKey(l), l]))
       const nextL = new Map(next.links.map(l => [linkKey(l), l]))
       return {
         addNodes: next.nodes.filter(n => !prevN.has(n.id)),
         updateNodes: next.nodes.filter(n => {
           const p = prevN.get(n.id)
           return p !== undefined && (p.label !== n.label || p.detail !== n.detail)
         }),
         removeNodeIds: prev.nodes.filter(n => !nextN.has(n.id)).map(n => n.id),
         addLinks: next.links.filter(l => !prevL.has(linkKey(l))),
         removeLinks: prev.links.filter(l => !nextL.has(linkKey(l))),
       }
     }
     ```
     Before coding, check the exact `Plan` indexing: `plans.versions[plans.approvedVersion - 1]`
     assumes 1-indexed versions — read how `fold` handles `plan_approved` in
     `packages/kernel/src/projections.ts` and index the same way (use an existing helper if one
     exists).
   - **Files:** `packages/ui-core/src/graph.test.ts` — pure fixtures (copy the event-constructor
     style from `packages/kernel/src/replay.test.ts` so payloads validate):
     - projects tasks, steps, artifacts, notes and their edges (assert node types + one
       `relates_to` note→note link)
     - never links to a node that does not exist: feed a note link to a deleted note and a task
       with a missing parent; assert every `links[].source/target` appears in `nodes[].id`
     - `diffGraphs`: appending one `memory_written` yields a patch whose `addNodes` is exactly
       `['note:project\u0000fresh-note']` and empty `removeNodeIds`
     - `diffGraphs`: a step status flip appears in `updateNodes`; a `memory_deleted` puts the
       note in `removeNodeIds` and its edges in `removeLinks`
   - **Verify:** `bun test packages/ui-core/src/graph.test.ts` green; typecheck clean.

6. **Step 2.3 — `ProjectSessions` (the live store every adapter subscribes to)**
   - **Why:** this is the heart of the adapter design: web SSE, the future TUI, and any other
     surface consume the SAME subscription API. No adapter ever talks to Postgres directly.
   - **Files:** `packages/ui-core/src/sessions.ts`
     ```ts
     import { openStorage, listProjectIds, fold, type State, type Storage } from '@orc/kernel'
     import type { EventRecord } from '@orc/contracts'
     import { buildGraph, diffGraphs, emptyPatch, type Graph, type GraphPatch } from './graph'

     export interface SessionSnapshot { seq: number; graph: Graph }
     export interface SessionUpdate { seq: number; patch: GraphPatch; event: EventRecord }
     export type Unsubscribe = () => void

     export interface ProjectSessions {
       projects(): Promise<Array<{ id: string; name: string | null }>>
       snapshot(projectId: string): Promise<SessionSnapshot>
       // fires per event whose patch is non-empty; raw event included so richer adapters
       // (TUI event feed, toasts) can show more than the graph delta
       subscribe(projectId: string, cb: (u: SessionUpdate) => void): Promise<Unsubscribe>
       // catch-up for resume: one cumulative patch from a client's seq to now
       since(projectId: string, fromSeq: number): Promise<SessionUpdate | null>
       nodeDetail(projectId: string, nodeId: string): Promise<unknown | null>
       close(): Promise<void>
     }

     export function createProjectSessions(opts: {
       url: string
       cwdProject?: { id: string; name: string }
     }): ProjectSessions { ... }
     ```
     Implementation per project (lazily created on first touch, cached):
     `events = await store.events.all()`; keep `{ events, state: fold(events), graph, seq }`;
     ONE `store.events.subscribe({ fromSeq: seq }, e => apply(e))`. `apply(e)`: push onto
     `events`, refold, `next = buildGraph(...)`, `patch = diffGraphs(graph, next)`, advance
     `graph/seq`, notify subscribers when `!emptyPatch(patch)`.
     `since(p, fromSeq)`: rebuild the graph at `fromSeq` by folding the cached events up to it,
     return `diffGraphs(thatGraph, current)` with the current seq (null if empty).
     `nodeDetail` switches on the node-id prefix:
     - task uuid → `{ task, usage: taskUsage(state, id), steps: [...state.steps.get(id)?.values() ?? []], artifacts: state.artifacts.get(id) ?? [] }`
     - `step:<taskId>:<stepId>` → the `StepState` + the plan step (instructions, modelRef, maxIterations)
     - `note:<scope>\u0000<id>` → full `MemoryWrittenPayload` + `hits` (count of `memory_accessed`
       events matching `{id, scope}`)
     - `artifact:<taskId>:<path>` → the `ArtifactRecord`
     - unknown → null
     `ponytail:` refold-per-event and fold-at-seq for `since` are O(events) — swap in an
     incremental kernel `applyEvent` behind this same interface when a log gets big.
   - **Files:** `packages/ui-core/src/sessions.test.ts` — against `createTestDb` (import the
     same helper `packages/cli/src/main.test.ts` uses): snapshot returns the appended task
     node; a subscriber receives a patch adding exactly a newly appended note; `since(oldSeq)`
     returns that note in one cumulative patch; `nodeDetail` returns the task title; `close()`
     unsubscribes cleanly (no dangling handles — test exits).
   - **Files:** `packages/ui-core/src/index.ts` — export everything above.
   - **Verify:** `bun test packages/ui-core` green.

### Phase 4 — the page (browser adapter behind `GraphRenderer`)

8. **Step 4.1 — renderer adapter + sigma implementation**
   - **Why:** the renderer is the third adapter seam: the page logic talks to an interface;
     sigma is implementation #1, cosmos is a later drop-in for 100k+ graphs.
   - **Files:** `packages/graph-ui/page/renderer.ts`
     ```ts
     import type { Graph, GraphPatch } from '@orc/ui-core'

     export interface GraphRenderer {
       setGraph(g: Graph): void            // full snapshot (project switch / initial load)
       applyPatch(p: GraphPatch): void     // incremental — MUST NOT recompute the world
       onNodeClick(cb: (nodeId: string) => void): void
       destroy(): void
     }
     ```
   - **Files:** `packages/graph-ui/page/sigma-renderer.ts` — `SigmaRenderer implements GraphRenderer`:
     graphology `Graph` as the store; `setGraph` = clear + `addNode/addEdge` (initial positions:
     random or circular — FA2 takes over); `applyPatch` = `addNode` (spawn new nodes at a linked
     neighbor's position + jitter so they grow out of their cluster, random if none) /
     `mergeNodeAttributes` / `dropNode` (drops incident edges automatically) / `addEdge`/`dropEdge`;
     ForceAtlas2 via `graphology-layout-forceatlas2/worker` (`FA2Layout`, `start()` on load,
     keep running while patches arrive, `stop()` after idle — e.g. 10s with no patch, restart on
     the next one); node color by `type` (task `#7aa2f7`, step `#9ece6a`, artifact `#e0af68`,
     note `#bb9af7`), size by degree (sigma reducers); click via sigma's `clickNode`.
   - **Files:** `packages/graph-ui/page/index.html` — minimal shell: `#graph` (flex-fill),
     `#panel` (right, 340px, scrollable), `#bar` with `<select id="project">`, dark theme,
     `<script type="module" src="./app.ts">` (Bun bundles TS + sigma/graphology from the HTML
     import).
   - **Files:** `packages/graph-ui/page/app.ts` — the page logic (renderer-agnostic):
     ```ts
     import type { GraphPatch } from '@orc/ui-core'
     import { SigmaRenderer } from './sigma-renderer'

     const renderer: GraphRenderer = new SigmaRenderer(document.getElementById('graph')!)
     let project: string | null = null
     let es: EventSource | undefined

     async function load(): Promise<void> {
       const g = await (await fetch(`/api/graph?project=${encodeURIComponent(project!)}`)).json()
       renderer.setGraph(g)
       watch(g.seq)
     }
     function watch(fromSeq: number): void {
       es?.close()
     ```
